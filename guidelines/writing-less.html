Writing LESS

Design team LESS/CSS guidelines on Stack Overflow

Goals

  • We’re all contributing code in the same or similar way through clearly defined guidelines. (Get on the same page, then we can build upon this more in the future.)
  • Keep the system accessible and simple to the design team and devs outside the design team.
  • Minimize unnecessary visual variation within the design of our product.

Guidelines

  • Code should be clear and easy to understand by anyone at a glance. The best code is easily maintainable.
  • Do not over abstract the code. Simplicity will allow multiple authors to contribute in a consistent manner, where complexity may cause confusion and fracturing.
  • If you’re going to make a change that impacts everyone, make sure you have a consensus from the team.
  • Componentize. Reuse. Standardize.

Naming conventions

Use dashes with no capitalization when you create new classes. You should almost never need to use IDs.

Class names

  • DO
    • .my-class { }
  • DON'T
    • .myClass { }
    • .my_class { }

Image names

Images names follow the same naming convention as classes. Image file names are prefixed with their usage.

  • DO
    • bg-header.png
    • icon-home.svg
    • bg-hero.jpg
  • DON'T
    • bgHeader.png
    • iconhome.png
    • bg_hero.png

Units

Color

When implementing feature styles, you should only be using color variables provided by _vars-colors.less.

When adding color variables use hexadecimal values. If an alpha is needed use LESS’s color functions RGBa.

  • DO
    • color: @orange;
    • color: fade(@orange, 90%);
  • DON'T
    • color: 244, 128, 36

Componentizing

Consistent use of patterns and style is a main reason the pattern library exists. Whenever possible, user interface elements should to be treated as generic components that can be used everywhere on the site.

Components should belong to their own less file. For example, all general button definitions should belong in _buttons.less.

Keep class names for modules generic. If you need to add specific styling, add a second area-specific class name, but this should be avoided if possible.

Name spacing

Name-spacing should be made at a descriptive, functional level. Not at a page location level. For example, .profile-header could become .header-hero-unit.

  • DO
    • .nav, .home-bar
  • DON'T
    • .nav, .home-nav

Code style

Comments

Use LESS style comments because they are compiled out. They should only be used to clarify code, add to-dos, and organization separate sections of the file.

  • DO
    • // Title of section
  • DON'T
    • /* Title of section */
    • ** Title of Section **

Variables

Global variables should go in their own top level .less file. If you’re creating a variable that is specific to a component then the variable should go within the component’s .less file.

Use variables only for the most important and reusable styles within the UI. If everything is important, then nothing is.

Variables should only be created for the following reasons:

1 There is a need to refer to the style many times in the .less files.

  • DO
    • @link-color
    • (link color is repeated in multiple places in different .less files)
  • DON'T
    • @button-padding-left
    • (this should be set once and doesn’t need to be reused)

2. The value of the property needs to vary in multiple contexts.

(This is particularly important within the Q&A sites because it allows us to easily adjust things like button colors.)
